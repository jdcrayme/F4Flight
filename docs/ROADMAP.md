# F4Flight — FreeFalcon Dynamic Campaign Mission Capability Roadmap

## 1. Executive Summary

The ultimate goal of the **F4Flight** project is to act as a modern, modular, thread-safe, and highly performant flight model and tactical AI engine that can natively replicate and fly **any mission role or flight profile** generated by the FreeFalcon/Falcon 4 Dynamic Campaign Engine.

Falcon 4's dynamic campaign engine operates on a hierarchical command-and-control simulation, issuing Air Tasking Orders (ATOs) to Packages and Flights. To execute these missions, the AI pilot must not only fly realistic flight dynamics but also exhibit complex tactical decision-making, manage sensors and weapons, communicate via radio, and dynamically coordinate with flight leads, wingmen, packages, air traffic control (ATC), and ground-controlled intercept (GCI/AWACS).

This document outlines the authoritative, multi-phase technical roadmap to bridge all remaining capability gaps, transition the AI architecture to a reactive state-tree framework, implement all 41 dynamic campaign mission types (`AMIS_*`), and establish a robust multi-aircraft package tactical simulation.

## 2. Current Capabilities Baseline

Through successive development sessions, F4Flight has evolved into a highly capable modern C++17 library. The current baseline features:
* **High-Fidelity 6-DOF Flight Model & FCS:** Validated performance across 17+ aircraft categories (F-16, F-15, F-18, F-22, MiG-29, Su-34, A-10, B-52, C-130).
* **AI-Level Gain Scheduling & Flight Phase Controls:** Dynamic, phase-aware PID tuning (Cruise, Combat, Formation, Approach, Flare, GroundOps) with integrated Phugoid damping and adaptive roll-rate damping.
* **Basic Autopilot & Navigation:** Altitude-hold, heading-select, altitude-select, and pitch/roll-hold.
* **Maneuver Primitives & BFM:** Pitch/Roll/GammaHold, Loiter, WaypointCapture, and RollAndPull (BFM).
* **3-Tier Cascade Testing & Visualization Framework:** 58 tests spanning Low-Level behaviors, High-Level chains, and End-to-End missions.
* **Tactical AI Foundations:**
  * **Air-to-Air:** BVR (Pursuit/Crank/Beam/Drag), WVR (RollAndPull), Merge, Collision Avoidance, and Missile/Guns Defeat.
  * **Air-to-Ground:** Dive-bomb, Level-delivery, and Toss/Loft attack profiles.
  * **Aerial Refueling (AAR):** Racetrack rendezvous, 4-phase refueling sequence (Approach, Precontact, Contact, Disconnect).
  * **Formation Station-Keeping:** 2-ship and 4-ship Wedge, Trail, Echelon, and Spread, using a robust PD lateral cross-track and along-track closure controller.
  * **Wingman Radio Comms:** Throttled radio-queue producing 20 distinct voice/display call types.
  * **Flight Lead Decisions:** Target scoring/priority, tactical engagement rules, and disengagement rules.

## 3. Dynamic Campaign Integration Architecture

To interface F4Flight with a dynamic campaign engine, we must establish a clear boundary between **Strategic Campaign Simulation** and **Tactical Flight Execution**.

```
  ┌────────────────────────────────────────────────────────┐
  │              DYNAMIC CAMPAIGN ENGINE (Host)            │
  │  - Strategic ATO / Package Coordinator                 │
  │  - Threat/Target database & Pathfinder                 │
  │  - ATC / GCI Simulation                                │
  └───────────────────────────┬────────────────────────────┘
                              │ API (C-Interface / gRPC)
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │                     F4FLIGHT ENGINE                    │
  │                                                        │
  │   Blackboard / Mission Task Queue                      │
  │   [ Task 1: Takeoff ] -> [ Task 2: Navigate ] -> ...   │
  │                                                        │
  │   ┌────────────────────────────────────────────────┐   │
  │   │          BEHAVIOR TREE (Tactical AI)           │   │
  │   │  - Safety & Defensive overrides                │   │
  │   │  - Offensive engagement & Role coordination   │   │
  └───────────────────────┬────────────────────────┘   │
                          │ Yaw/Pitch/Roll/Throttle    │
                          ▼                            │
  ┌────────────────────────────────────────────────┐   │
  │          FLIGHT MODEL & AUTOPILOT FCS          │   │
  │  - 6-DOF Equations of Motion                   │   │
  └────────────────────────────────────────────────┘   │
  └────────────────────────────────────────────────────────┘
```

The campaign integration layer is driven by the **Blackboard**:
1. **ATO Task Injection:** The host maps campaign mission parameters into a sequential queue of `MissionTask` nodes (e.g., Takeoff on Runway 15, Ingress Waypoints, CAP Loiter Orbit, Weapon Release coordinates, RTB divert field) loaded onto the Blackboard's `FlightPlan` structure.
2. **Dynamic GCI Retasking:** When a pop-up threat is detected, GCI sends a message. The F4Flight Blackboard immediately prepends or swaps tasks in the flight plan. The reactive Behavior Tree automatically shifts execution from the active task (e.g., CAP) to an `Intercept` task without breaking flight loop continuity.
3. **Environment Syncing:** Every frame, the host provides `FrameInputs` (the current truth sensor picture, friendly coordinate lists, threat coordinates, fuel rates, and damage states), while F4Flight returns the updated `AircraftState` and `PilotInput` (along with active radio queues, weapon release consents, and ATC messages).

## 4. Phase-by-Phase Roadmap

### Phase 1: Reactive State Tree (Behavior Tree / HSM) Refactoring
*Current State: High-level tactical design docs exist in `docs/design/`, but the AI's core routing remains a flat, priority-based priority-stack (`resolveMode`).*

To support complex, nested, and stateful behaviors (such as transitioning from defensive maneuvering back to a flight plan, or managing wingman-specific flight tasks), we must implement the hybrid Behavior Tree (BT) and Hierarchical State Machine (HSM) architecture.

#### Key Objectives:
1. **Bootstrap Core BT Types:**
   * Implement `BehaviorNode` (with explicit `onEnter`, `onTick`, `onExit` lifecycles).
   * Implement control flow nodes: `SelectorNode` (OR logic), `SequenceNode` (AND logic), and `DecoratorNode` (conditional filtering).
   * Implement `Blackboard` holding world parameters, ownship state, target pointers, role structures, and the active `FlightPlan`.
2. **Wrap Legacy Primitives in Action Nodes:**
   * Wrap existing behaviors (e.g., `runWaypoint`, `runRefueling`, `runGroundAttack`, `runWVREngage`) into leaf `ActionNode` subclasses that execute them cleanly using Blackboard context.
3. **Implement the "Shadow Executive" validation pipeline:**
   * Run the Behavior Tree side-by-side with the legacy priority-stack.
   * Compare `activeMode()` with `blackboard.state->expectedModeFromBT` and log any discrepancies.
   * Run all 58 existing scenarios under shadow execution. Once 100% of scenarios match exactly, **flip the switch** to make the Behavior Tree the production executor.
4. **Clean up Steering Controller:**
   * Move from the monolithic `compute()` method to a clean, modular Behavior Tree traversal, and deprecate the legacy priority-stack and `resolveMode()`.

---

### Phase 2: Comprehensive Air-to-Ground (A/G) Strike & SEAD Systems
*Current State: Basic A/G supports Dive-bomb, Level-delivery, and Toss-bomb profiles. There is no targeting pod management, no SEAD radar tracking, and no forward-air-control logic.*

To fly dynamic campaign A/G missions, the AI must handle advanced target acquisition, standoff weapons, anti-radiation targeting, and cooperative ground control.

#### Key Objectives:
1. **Targeting Pod (TGP) & Sensor Management:**
   * Implement an A/G radar mode and TGP tracker within `SensorFusion`.
   * Simulate terrain elevation lookups, laser target designation, and target track preservation.
2. **Standoff Weapon Delivery Profiles:**
   * **AGM-65 Maverick:** Implement stand-off acquisition, seeker lock-on range calculations, and lock-on times based on terrain cover.
   * **JDAM/JSOW (GPS/INS):** Implement high-altitude level-release profiles within the weapon's kinematic launch envelope (LAR - Launch Acceptability Region).
3. **Suppression of Enemy Air Defenses (SEAD) Logic:**
   * Map radar-warning receiver (RWR) emitters to `SensorContact` entries.
   * **AGM-88 HARM:** Implement HARM Targeting System (HTS) profiles: Pre-Briefed (PB), Target-Of-Opportunity (TOO), and Self-Protect (SP).
   * Create reactive SEAD flight profiles where the aircraft orbits outside the SAM engagement ring, detects emission, lofts a HARM, and performs defensive beam-maneuvers.
4. **Forward Air Controller (FAC) AI (`facbrain.cpp` port):**
   * Implement FAC-to-Strike coordination: FAC orbits near target, detects ground threats, generates coordinate messages, and directs the Strike flight to the target area via standard 9-line briefings.

---

### Phase 3: Multi-Ship Flight Coordination & Advanced Formations
*Current State: Flight Lead logic exists but is limited to target selection. Wingmen fly simple formation or follow hardcoded orders. There is no dynamic team sorting or division split.*

Dynamic campaign combat requires multi-ship packages to work in concert, dividing targets, protecting each other, and executing complex tactical geometry.

#### Key Objectives:
1. **Dynamic Target Sorting:**
   * Implement tactical target sorting for 2-ship and 4-ship flights.
   * Use the "radar sort" algorithm: Lead locks the target furthest to the left (or highest altitude), Wingman locks the rightmost, ensuring zero redundant weapon launches on the same target.
2. **Tactical Division Splits & Element Tactics:**
   * **Bracket Maneuver:** Lead and Wingman split laterally, forcing an inbound target to commit to one, allowing the other to execute a flanking flank-shot.
   * **Pincer (Pince) Maneuver:** Enveloping an inbound bandit using coordinated, opposing turns.
   * **Drag-and-Bag / Beam Tactics:** Coordinating level turns where one flight member acts as bait (dragging the enemy) while the other climbs to execute a rear-aspect intercept.
3. **Promotable / Failover Wingman Roles:**
   * Implement blackboard-driven role transitions. If the Flight Lead (`leadId`) is destroyed or suffers a critical systems failure, the Wingman is automatically promoted to Lead, takes ownership of the package flight plan, and commands remaining elements.
4. **Advanced Formation Maneuvers:**
   * Implement tactical turns (Cross Turn, Hook Turn, Delayed Turn, In-Place Turn) where wingmen cross-under or adjust spacing dynamically during bank angles.

---

### Phase 4: Support Roles & Specialized Airspace Operations
*Current State: Basic AAR refueling sequence is implemented, but there is no queue management, tanker pattern management, or complex ATC airspace operations.*

Support missions form the operational backbone of a dynamic campaign. F4Flight must support flying both receivers and support providers (tankers, AWACS) inside the simulated airspace.

#### Key Objectives:
1. **Tanker AI Queue Management (`tankbrn.cpp` port):**
   * Fly precise AAR racetrack patterns using autopilot altitude/heading hold.
   * Manage the active refueling queue: accept join-up requests, assign "left contact anchor" positions to waiting flights, clear individual aircraft to the "pre-contact" slot, track fuel transfer, and command disconnects.
2. **AWACS / GCI Airspace Scanning (`AMIS_AWACS`):**
   * Orbit in a safe, high-altitude CAP zone.
   * Process complete theater-level radar pictures, filter out friendly flights, and broadcast vector-intercept messages (Bullseye calls) to active friendly sweeps/intercepts.
3. **Advanced ATC & Terminal Airspace Management:**
   * Coordinate with airfield ATC for multi-ship landing sequences.
   * Implement overhead breaks (tactical recovery), pattern entries (Downwind, Base, Final), spacing management, and missed-approach/go-around flight paths.
   * Support taxi-way holds, runway queuing, and takeoff clearances.
